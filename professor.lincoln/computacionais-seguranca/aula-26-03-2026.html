BIOS: Basic Input/Output System

Legacy Mode

O que é?

Firmware armazenado em um chip ROM/Flash na placa-mãe. É o primeiro código executado ao ligar o hardware (Cold Boot).

Limitações Críticas:

  • Modo Real (16 bits): Acesso limitado à memória RAM (1MB).
  • MBR (Master Boot Record): Suporta apenas discos de até 2TB e 4 partições primárias.
  • Interface: Apenas teclado, sem suporte a rede ou gráficos avançados no boot.

Fluxo de Execução (POST)

1. Power On
2. CPU Init (Reset Vector)
3. POST (Diagnóstico de Hardware)
4. Leitura do Setor 0 (MBR)
5. Salto para o Bootloader do SO

"A BIOS não entende sistemas de arquivos, ela apenas lê setores físicos."

A Modernidade

UEFI: Unified Extensible Firmware Interface

Arquitetura 32/64 bits

Permite drivers independentes de CPU e inicialização muito mais veloz.

Suporte GPT

Discos de até 9.4 Zettabytes e partições virtualmente ilimitadas.

🛡️ Secure Boot

Pilar de Segurança: Verifica a assinatura digital do Bootloader. Impede que Rootkits iniciem antes do SO.

Nota de Segurança: Conforme Stallings (2012), a UEFI permite que o firmware tenha sua própria pilha de rede, possibilitando diagnósticos remotos e atualizações de segurança mesmo sem um HD instalado.

Pilar de Segurança

Secure Boot

A Cadeia de Custódia do Software

Como Funciona?

O Firmware UEFI valida a Assinatura Digital do carregador de inicialização (Bootloader) contra um banco de chaves confiáveis armazenado no hardware.

  • ✓ Apenas SOs autorizados iniciam.
  • ✓ Bloqueia códigos não assinados.
  • ✓ Protege a integridade do Kernel.

"A segurança do sistema operacional é irrelevante se o processo de boot puder ser subvertido antes mesmo do kernel carregar suas políticas de proteção."

— Stallings (2012)

Rootkits & Bootkits

Ameaças Invisíveis ao Sistema Operacional

O que são?

Malwares projetados para infectar o MBR/VBR ou o próprio Firmware. Eles "sobem" antes do antivírus.

O Perigo

Podem interceptar chamadas do Sistema Operacional, esconder arquivos e processos, tornando-se persistentes mesmo após a formatação do SO.

Conexão A3: Ao projetar um ambiente seguro, a ativação do Secure Boot e a configuração de Senhas de Supervisor no Setup são os primeiros passos do Hardening de infraestrutura.

Kernel & Drivers

O Kernel

É a parte "residente" do SO. Gerencia:

  • Acesso à Memória RAM (Proteção).
  • Escalonamento da CPU.
  • Chamadas de Sistema (System Calls).

Os Drivers

Atuam como "tradutores" específicos. Sem o driver correto, o Kernel não sabe como operar o hardware físico.

⚠️ Falhas em drivers são a causa principal de interrupções críticas (Tela Azul).

"O kernel utiliza drivers para abstrair o hardware, permitindo que o mesmo código de aplicação funcione em periféricos de diferentes fabricantes."

— Tanenbaum (2007)

Gerenciamento de Recursos

CPU & Memória

Processador (CPU)

O SO decide quem "usa a vez".

  • Escalonamento: Alternância rápida entre processos (Ilusão de paralelismo).
  • Prioridades: Processos críticos do sistema vs. Aplicações de usuário.

Memória RAM

O SO atua como um "corretor de imóveis".

  • Isolamento: Garante que o App A não leia dados do App B.
  • Memória Virtual: Uso do disco para "fingir" ter mais RAM disponível.

"Sem o gerenciamento de memória do kernel, um único erro de programação em uma aplicação poderia corromper todo o sistema."

— Machado & Maia (2013)

Gerenciamento de E/S

A Comunicação com o Mundo Externo

Interrupções

O dispositivo avisa a CPU que terminou uma tarefa ou precisa de atenção imediata.

Buffering & Caching

O SO armazena dados temporariamente para compensar a diferença de velocidade entre dispositivos.

Spooling

Gerenciamento de filas (ex: impressoras) para que múltiplos processos não conflitem no mesmo recurso.

💡 Foco em Segurança: O SO deve validar toda entrada de periféricos para evitar ataques de injeção de hardware (BadUSB).

Momento de Refresh: Assim como a DRAM precisa de recarga para não perder dados, o cérebro precisa de pausa para consolidar a Arquitetura de Sistemas.

Mídias e Imagens ISO

Segurança de Origem

A Regra de Ouro do Analista:

Sistemas Operacionais devem ser baixados apenas de fontes oficiais (Microsoft, Debian, Canonical, etc.).

⚠️ O Perigo das ISOs Modificadas: Versões "Lite" ou "Otimizadas" de terceiros podem conter Backdoors e Keyloggers pré-instalados no Kernel.

Por que ISOs Estáticas são um risco?

Vulnerabilidades de Dia Zero (0-Day) podem surgir após a criação da mídia. A instalação offline deixa o sistema exposto até o primeiro ciclo de update.

— Ref: Machado & Maia (2013)

Verificação de Integridade

A Assinatura Matemática do Arquivo

Como garantir que a ISO não foi corrompida no download ou alterada por um atacante (Man-in-the-Middle)? Usamos algoritmos de Hash (SHA-256).

# Exemplo de verificação via CLI (Linux/PowerShell)

Get-FileHash ./ubuntu-24.04.iso -Algorithm SHA256

Princípio da Segurança: Se um único bit for alterado na ISO original, o Hash gerado será completamente diferente.

"A integridade dos dados assegura que a informação não foi modificada de forma não autorizada desde sua criação."

— Stallings (2012)

MBR: Master Boot Record

Padrão 1983

Características e Limites:

  • 📍 Localização: Reside no primeiro setor do disco (Setor 0).
  • 📏 Capacidade: Limite máximo de 2.2 Terabytes.
  • 🧩 Partições: Máximo de 4 partições primárias.
  • ⚠️ Risco: Se o setor do MBR corromper, o disco inteiro fica inacessível (Ponto Único de Falha).

"O MBR funciona em conjunto com a BIOS Legacy. Ele contém a Tabela de Partição e o código de inicialização que aponta para o Sistema Operacional."

— Ref: Stallings (2012)

GPT: GUID Partition Table

O Parceiro Nativo do UEFI

Escalabilidade

Suporta discos de até 9.4 Zettabytes e permite até 128 partições primárias nativamente.

Resiliência (Redundância)

Armazena cópias da tabela de partição no início e no fim do disco. Recuperação automática em caso de erro.

Dica Técnica: Para instalar sistemas modernos (Windows 11) com Secure Boot ativo, o disco deve estar em GPT.

Ameaças de "Dia Zero"

Risco Crítico

O que é um 0-Day?

É uma vulnerabilidade de software que é explorada por atacantes antes que o fabricante tenha conhecimento dela ou lance uma correção.

Por que na instalação? ISOs são estáticas. Uma mídia de 6 meses atrás está exposta a todas as vulnerabilidades descobertas nesse intervalo.

"O tempo entre a descoberta de uma vulnerabilidade e a aplicação do patch é a janela de oportunidade mais valiosa para um intruso."

— Machado & Maia (2013)

Hardening Inicial

Protocolo de Pós-Instalação

1. Patch Management

Executar o ciclo completo de atualizações (Update) antes de qualquer outra configuração.

2. Desativação de Serviços

Encerrar protocolos legados e serviços desnecessários para reduzir a Superfície de Ataque.

3. Políticas de Conta

Renomear contas de administrador, forçar senhas fortes e ativar MFA onde disponível.

Conexão A3: Um relatório técnico de SO deve listar quais serviços foram desativados para justificar a segurança da arquitetura proposta.

Logs de Eventos

Auditoria & Diagnóstico

O que são?

Registros históricos gerados pelo Kernel, drivers e aplicações. São a primeira fonte de dados para o Troubleshooting.

Categorias Principais:

  • 🔴 Erro: Falha crítica (ex: Driver não carregado).
  • 🟡 Aviso: Problema potencial (ex: Disco quase cheio).
  • 🔵 Informação: Operação bem-sucedida (ex: Serviço iniciado).

Visão Proativa:

Muitas vezes o sistema inicia, mas o log mostra que um recurso (como aceleração de hardware) foi desativado por erro. O log permite a correção proativa antes do usuário reclamar.

— Ref: Machado & Maia (2013)

O Fator Tempo (NTP)

Por que relógios errados quebram a segurança?

O Caos na Auditoria

Se um ataque ocorre às 22h e o log registra 15h por fuso horário ou relógio dessincronizado, a correlação de eventos com firewalls e outros servidores torna-se impossível.

"A evidência digital só tem valor legal e técnico se puder ser situada com precisão em uma linha do tempo confiável."

— Stallings (2012)

Insight Lab: No laboratório, verifique se o serviço de tempo está sincronizado com a.ntp.br antes de iniciar a análise de logs.

11. Prática de Laboratório

Instalação Otimizada

Configuração de Firmware

  • Ativar Secure Boot (Modo UEFI).
  • Habilitar Virtualização (VT-x / AMD-V).
  • Definir a Ordem de Boot (Mídia Oficial Primeiro).
  • Ativar TPM 2.0 (Se Windows 11).

Estratégia de Disco

  • 🗂️ Tabela: Converter disco para GPT.
  • 🛡️ Partição: Criar partição de Sistema (EFI).
  • 🚀 Swap/Paging: Definir tamanho fixo (Otimização).
  • 🧹 Limpeza: Remover partições residuais de mídias anteriores.

Checklist de Entrega

Status de Configuração:

  • [ ] SO instalado e Kernel reconhecendo Hardware.
  • [ ] Drivers críticos atualizados (Chipset/Rede).
  • [ ] Ciclo de Windows Update / Apt Upgrade finalizado.
  • [ ] Relógio sincronizado via NTP (a.ntp.br).
  • [ ] Log de Eventos verificado (Sem Erros Críticos).
Duração: 60 minutos
