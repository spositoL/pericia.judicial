O Ponto de Partida

Arquitetura Monolítica

Unidade Única: UI, Business Logic e Data Access em um só binário.

Risco: Uma falha em um módulo derruba o sistema inteiro (Tanenbaum, 2007).

  • ❌ Escalabilidade Ineficiente
  • ❌ Deploy Arriscado
  • ❌ Barreira Tecnológica

A Solução Distribuída

Microsserviços

"Componentização de sistemas onde cada serviço é um processo independente, comunicando-se via mecanismos leves (APIs)."
— Monteiro et al. (2020)

✅ Deploy Independente
✅ Tolerância a Falhas
✅ Stack Poliglota

Teoria de Sistemas

Desacoplamento

"A capacidade de alterar a implementação de um serviço sem afetar seus clientes."
— Coulouris et al. (2013)

Foco na Interface

O que importa é o contrato, não como o código foi escrito internamente.

Independência

Serviços podem ser reiniciados ou atualizados isoladamente (Tanenbaum, 2007).

Princípio da Responsabilidade

Alta Coesão

"Cada microsserviço deve ter uma única e clara razão para existir."

  • 🎯 Facilita a compreensão e manutenção do código.
  • 🎯 Reduz efeitos colaterais em mudanças de requisitos.
  • 🎯 Base para o Bounded Context (Contexto Delimitado).

Arquitetura de Entrada

API Gateway

"Um ponto único de entrada que encapsula a arquitetura interna do sistema, oferecendo uma API sob medida para cada cliente."
— Adaptado de Monteiro et al. (2020)

O Acordo entre Serviços

Design de Interfaces

"Interfaces bem definidas permitem que sistemas heterogêneos operem como uma unidade coesa."
— Coulouris et al. (2013)

Ocultação de Detalhes

O cliente não precisa saber qual banco de dados ou linguagem o serviço usa.

Independência

A implementação interna pode mudar, desde que o contrato seja mantido.

Implementação Moderna

Contratos de API

API como Produto

  • 📜 Documentação Viva: Swagger / OpenAPI (Monteiro, 2020).
  • 📜 Versionamento: `/v1/`, `/v2/` para evitar quebras.
  • 📜 Idempotência: Garantia de estado em redes instáveis.

Roteamento

Mão na Massa

Atividades Práticas

1. Subir Microserviços (A e B)

Utilizar ambientes isolados para simular a distribuição (Monteiro, 2020).

2. Configurar o API Gateway

Criar rotas `/servico-a` e `/servico-b` no ponto único de entrada.

3. Testar a Interoperabilidade

Validar se o Serviço A consegue disparar uma ação no Serviço B via Gateway.

Unidade Curricular: Projeto A3

Ecossistema Distribuído

"O sucesso de um sistema distribuído reside na sua capacidade de escalar horizontalmente e manter a disponibilidade sob carga."
— Baseado em Monteiro et al. (2020)

🚀 Requisito Técnico

O sistema não pode ser um monólito. Deve haver separação clara entre ao menos dois serviços e um Gateway.

🛡️ Requisito de Gestão

Uso de protocolos de comunicação padronizados para garantir a interoperabilidade (Coulouris, 2013).

O que será avaliado

Checklist Técnico

  • [ ]
    Contratos: Documentação mínima das rotas (Swagger ou similar).
  • [ ]
    Segurança: Demonstração de acesso controlado via Gateway.
